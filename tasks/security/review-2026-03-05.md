### [High] S-001: POST /api/employees에 RBAC 검증 누락
- **위치**: `app/api/employees/route.ts:39-51` (POST 핸들러)
- **설명**: POST 핸들러에서 user 존재만 확인하고 `user.role !== "ADMIN"` 검증이 없음
  ```typescript
  // 현재: 인증만 확인
  if (!user) return NextResponse.json({ error: "인증이 필요합니다." }, { status: 401 });
  // 역할 검증 누락!

  const body = await request.json();
  const { name, department, email } = body;
  // ... 조직원 생성 진행
  ```
- **영향**: 일반 사용자(USER role)가 API를 직접 호출하여 조직원을 무단 등록할 수 있음
  - 시나리오: `POST /api/employees { name: "신규직원", department: "임시" }` → 미인증된 데이터 추가
- **근거 표준**:
  - ISO 27001 A.9.1.2 (접근 통제)
  - ISMS-P 2.6.1 (사용자별 권한 관리)
- **Zero Trust 위반**: Least Privilege (API 수준에서 권한 검증 누락)
- **수정 방안**:
  ```typescript
  // app/api/employees/route.ts:41 추가
  if (user.role !== "ADMIN") {
    return NextResponse.json({ error: "권한이 없습니다." }, { status: 403 });
  }
  ```
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [High] S-002: 로그인 실패 감사 로그의 actor 필드 처리 미흡
- **위치**: `app/api/auth/login/route.ts:42-50` (로그인 실패 로그)
- **설명**: 로그인 실패 시 감사 로그를 기록하지만, actor 필드에 값을 넣지 않음 (username이 검증되지 않았으므로)
  ```typescript
  // 현재: actor 필드 없음
  await prisma.auditLog.create({
    data: {
      entityType: "USER",
      entityId: 0,  // ← 잘못된 ID
      action: "LOGIN_FAILED",
      details: JSON.stringify({ username, ip }),
      // actor 필드 누락 — 행위자 식별 불가
    },
  }).catch(() => {});
  ```
- **영향**:
  - 로그인 실패 사건 추적 시 행위자(username)를 식별할 수 없음
  - Brute-force 공격 탐지 시 공격자 이름 기록 불가
  - ISMS-P 감사 로그 요구사항 미충족
- **근거 표준**:
  - ISO 27001 A.12.4.1 (감사 로그 기록 요구)
  - ISMS-P 2.11 (로그인 사건 로깅)
- **Zero Trust 위반**: Assume Breach (침해 추적 불가)
- **수정 방안**:
  ```typescript
  // app/api/auth/login/route.ts:42-50
  // 시도한 username을 details가 아닌 actor/notes 필드에 기록
  await prisma.auditLog.create({
    data: {
      entityType: "USER",
      entityId: 0,
      action: "LOGIN_FAILED",
      actor: username,  // ← 시도한 username 기록
      details: JSON.stringify({ ip, reason: "invalid_credentials" }),
    },
  }).catch(() => {});
  ```
- **담당 세션**: 백엔드
- **상태**: 🟡 부분 수정됨 (actor 필드 추가 필요)

### [High] S-003: 로그인 실패 로그의 entityId=0은 부적절
- **위치**: `app/api/auth/login/route.ts:46`
- **설명**: 로그인 실패 시 해당 사용자가 DB에 없으므로 `entityId: 0`으로 기록
  - 이는 실제 User 테이블의 ID와 충돌 가능성 존재
  - 감사 로그 추적 시 혼동 유발
- **영향**: 감사 로그 분석 시 유효하지 않은 User 참조 → 연관성 추적 불가
- **근거 표준**: ISMS-P 2.11 (감사 로그 정확성)
- **수정 방안**: `entityId: null` 또는 음수(-1) 사용
  ```typescript
  await prisma.auditLog.create({
    data: {
      entityType: "USER",
      entityId: -1,  // ← 미등록 사용자 구분
      action: "LOGIN_FAILED",
      actor: username,
      details: JSON.stringify({ ip }),
    },
  }).catch(() => {});
  ```
- **담당 세션**: 백엔드
- **상태**: 🟡 부분 수정 (entityId 값 검토 필요)

### [Medium] S-004: 로그아웃 후에도 세션 쿠키 재사용 가능성
- **위치**: `app/api/auth/logout/route.ts:31-37`
- **설명**: 로그아웃 시 쿠키를 `maxAge: 0`으로 설정하여 클라이언트에서 삭제하지만, 서버 측에서 세션 유효성 검증 필요
- **현재 구현**:
  ```typescript
  response.cookies.set(SESSION_COOKIE, "", {
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax",
    path: "/",
    maxAge: 0,  // ← 클라이언트 삭제만 의존
  });
  ```
- **영향**:
  - 클라이언트가 쿠키를 조작하면 로그아웃된 세션 재사용 가능
  - 현재 `deleteSession(token)` 호출로 부분 완화되나, 에러 시 세션이 남음
- **근거 표준**: ISO 27001 A.9.4.4 (세션 관리)
- **수정 방안**: `deleteSession` 실패 시 에러 처리 강화
  ```typescript
  if (token) {
    try {
      await deleteSession(token);
    } catch (error) {
      console.error("Failed to delete session:", error);
      // 강제 로그아웃 처리 (쿠키 삭제 유지)
    }
  }
  ```
- **담당 세션**: 백엔드
- **상태**: 🟡 부분 개선 (에러 처리 추가 필요)

### [Info] S-006: X-Forwarded-For IP 파싱 안전성
- **위치**: `app/api/auth/login/route.ts:17`
- **설명**: IP 파싱 시 프록시 체인 처리
  ```typescript
  const ip = request.headers.get("x-forwarded-for")?.split(",")[0].trim() ?? "unknown";
  ```
  - 프록시가 정상이면 OK, 조작되면 IP 스푸핑 가능
- **현재 상태**: 폐쇄망 환경에서는 낮은 위험
- **개선 방안**: 프록시 체인 검증
  ```typescript
  const forwardedFor = request.headers.get("x-forwarded-for");
  const ip = forwardedFor ? forwardedFor.split(",").pop()?.trim() : "unknown";
  // 또는 신뢰할 수 있는 프록시만 allow 리스트로 관리
  ```
- **상태**: 🟢 수용 가능 (폐쇄망 보완)
