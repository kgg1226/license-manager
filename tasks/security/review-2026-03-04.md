## 발견 사항

### [High] F-001: API 라우트 전반 RBAC(역할 기반 접근 제어) 미적용

- **위치**: 아래 API 라우트 전체
  - `app/api/employees/[id]/route.ts` — PUT, PATCH, DELETE
  - `app/api/licenses/route.ts` — POST
  - `app/api/licenses/[id]/route.ts` — PUT, DELETE
  - `app/api/assignments/[id]/route.ts` — PUT, DELETE
  - `app/api/groups/route.ts` — POST
  - `app/api/groups/[id]/route.ts` — PUT, DELETE
  - `app/api/org/companies/route.ts` — POST
  - `app/api/org/units/route.ts` — POST (미확인)
- **설명**: 모든 변경 API 라우트에서 `getCurrentUser()`로 인증만 확인하고 `user.role`을 검증하지 않는다. `requireAdmin()`이 lib/auth.ts에 존재하나 API 라우트에서는 전혀 사용되지 않고 `app/admin/` Server Actions에서만 사용된다.
- **영향**: ROLE=USER인 일반 계정이 API를 직접 호출해 라이선스 생성·수정·삭제, 직원 삭제, 할당 취소, 그룹 변경 등 위험한 데이터 조작 가능. 관리자 전용 기능이 UI에서만 가려져 있고 서버 측 검증이 없는 구조.
- **근거 표준**: ISO 27001 A.9.1.2 (정보에 대한 접근), A.9.4.1 (정보 접근 제한), ISMS-P 2.6.1 (접근 통제 정책), ISMS-P 2.6.2 (정보시스템 접근)
- **Zero Trust 위반**: Least Privilege (최소 권한 원칙 — 기본값이 "허용"으로 설정된 구조)
- **수정 방안**:
  - 데이터 변경 API(POST/PUT/PATCH/DELETE)에 `requireAdmin()` 또는 역할 확인 코드 추가
  - 예시: `const user = await getCurrentUser(); if (!user || user.role !== "ADMIN") return NextResponse.json({ error: "권한이 없습니다." }, { status: 403 });`
  - 또는 API Route마다 `requireAdmin()`을 import하여 최상단에서 호출
  - 읽기 전용(GET) API는 인증된 사용자 전체 허용 유지 가능
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [High] F-002: Server Actions 인증 누락

- **위치**:
  - `app/employees/new/actions.ts:createEmployee` (전체 함수)
  - `app/settings/import/actions.ts:importCsv` (전체 함수)
- **설명**: 두 Server Action에 `getCurrentUser()` 또는 `requireAdmin()` 호출이 없다. Next.js Server Actions는 `/_next/action` HTTP 엔드포인트로 직접 POST 요청 가능하므로, 미인증 사용자도 직원 생성·CSV 대량 임포트 실행 가능.
  - `createEmployee`: 인증 없이 직원 생성 및 자동 라이선스 할당
  - `importCsv`: 인증 없이 라이선스·직원·배정 데이터 대량 생성/수정 가능 (5MB CSV 허용)
- **영향**: 미인증 또는 저권한 사용자가 직원 등록 및 대량 데이터 무단 조작 가능.
- **근거 표준**: ISO 27001 A.9.4.1, ISMS-P 2.5.1 (사용자 인증), ISMS-P 2.6.1
- **Zero Trust 위반**: Verify Explicitly (모든 요청에 인증 확인 불이행)
- **수정 방안**:
  - `createEmployee` 최상단에 인증 확인 추가
  - `importCsv`는 대량 데이터 변경 작업이므로 `requireAdmin()` 적용 권장
  - 예시: `const user = await getCurrentUser(); if (!user) throw new Error("인증이 필요합니다.");`
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [High] F-003: 로그인 실패 횟수 제한(Rate Limiting/Brute-force 방어) 없음

- **위치**: `app/api/auth/login/route.ts` 전체
- **설명**: 로그인 실패 시 아무런 횟수 제한, 지연 처리, 계정 잠금이 없다. 동일 IP 또는 계정명에 대해 무제한 시도 가능.
- **영향**: 자동화 도구를 이용한 비밀번호 브루트포스 공격에 무방비. 특히 단순 비밀번호를 사용하는 계정은 단시간 내 탈취 가능.
- **근거 표준**: ISO 27001 A.9.4.2 (보안 로그온 절차), ISMS-P 2.5.2 (사용자 식별), ISMS-P 2.5.4 (접근 실패 처리)
- **Zero Trust 위반**: Verify Explicitly (반복 실패 시 검증 강화 미적용)
- **수정 방안**:
  - 단기 해결: IP 기반 또는 계정명 기반 실패 횟수를 Redis/메모리에 저장하여 5회 실패 시 15분 잠금
  - 근본 해결: Rate Limiting 미들웨어 적용 (예: `next-rate-limit` 또는 직접 구현)
  - 실패 이벤트 AuditLog 또는 서버 로그에 기록 (username, IP, timestamp)
  - 폐쇄망 환경이지만 내부자 위협도 고려 필요
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [Medium] F-004: Audit Log actor 필드 누락 — 행위자 추적 불가

- **위치**: 아래 `writeAuditLog()` 호출 전체
  - `lib/assignment-actions.ts` (assignLicenses, unassignLicenses 내 다수)
  - `app/licenses/new/actions.ts:createLicense`
  - `app/licenses/[id]/edit/actions.ts` (updateLicense, deleteLicense)
  - `app/settings/import/actions.ts` (importLicenses, importEmployees 등)
- **설명**: `writeAuditLog()`의 `actor?: string` 파라미터가 정의되어 있으나 모든 호출 지점에서 전달하지 않아 DB의 `AuditLog.actor`가 항상 null로 저장된다. 어떤 사용자가 어떤 행동을 했는지 로그에서 식별 불가.
- **영향**: 보안 사고 발생 시 책임 추적(accountability) 불가. 내부자 위협 감지 불가. ISMS-P 감사 요건 미충족.
- **근거 표준**: ISO 27001 A.12.4.1 (이벤트 로깅), ISO 27701 8.2.5 (감사 로깅), ISMS-P 2.11.1 (로그 및 접속기록 관리)
- **Zero Trust 위반**: Assume Breach (침해 후 추적 불가)
- **수정 방안**:
  - Server Actions에서 `getCurrentUser()`를 호출하여 현재 사용자명을 `actor`로 전달
  - 예시: `await writeAuditLog(tx, { ..., actor: currentUser.username })`
  - Server Actions에서 `getCurrentUser()` 호출 시 "use server" 컨텍스트에서 cookies()를 통해 세션 정보 조회 가능
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [Medium] F-005: 인증 이벤트(로그인/로그아웃) 로깅 없음

- **위치**:
  - `app/api/auth/login/route.ts`
  - `app/api/auth/logout/route.ts` (파일 내용 미확인이나 동일 패턴 추정)
- **설명**: 로그인 성공/실패, 로그아웃 이벤트가 AuditLog 테이블이나 별도 보안 로그에 기록되지 않는다. `console.error`는 프로세스 재시작 시 사라지며 영속성 없음.
- **영향**: 계정 탈취 시도, 비정상 접근 패턴 사후 분석 불가. ISMS-P 필수 로그 항목 누락.
- **근거 표준**: ISO 27001 A.12.4.1, ISMS-P 2.11.1 (로그인 성공/실패 기록 의무), ISO 27701 8.2.5
- **Zero Trust 위반**: Assume Breach (침해 감지·추적 불가)
- **수정 방안**:
  - 로그인 성공: `AuditLog.create({ entityType: "USER", action: "LOGIN", actor: username, details: { ip: ... } })`
  - 로그인 실패: `AuditLog.create({ entityType: "USER", action: "LOGIN_FAILED", details: { username, ip: ... } })`
  - 로그아웃: `AuditLog.create({ entityType: "USER", action: "LOGOUT", actor: username })`
  - IP는 `request.headers.get("x-forwarded-for") || request.ip` 로 수집
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [Medium] F-006: Docker 컨테이너 root 사용자 실행

- **위치**: `dockerfile` (USER 지시문 없음)
- **설명**: `FROM node:20-alpine`은 기본적으로 root 사용자로 실행된다. `dockerfile`에 `USER` 지시문이 없어 프로덕션 컨테이너가 root 권한으로 동작한다.
- **영향**: 컨테이너 내 취약점(예: prototype pollution, RCE) 악용 시 root 권한으로 호스트 파일시스템 접근 가능. `volumes`로 마운트된 DB 파일에 root 권한으로 직접 접근·변조 가능.
- **근거 표준**: ISO 27001 A.12.6.1 (기술적 취약점 관리), ISMS-P 2.10.3 (운영 환경 보안)
- **Zero Trust 위반**: Least Privilege (컨테이너 프로세스에 과도한 권한 부여)
- **수정 방안**:
  - `dockerfile`의 runner stage에 비root 사용자 추가:
    ```dockerfile
    RUN addgroup -S appgroup && adduser -S appuser -G appgroup
    USER appuser
    ```
  - 단, `better-sqlite3` 네이티브 바이너리 및 DB 파일 볼륨 권한 호환성 테스트 필요
  - 볼륨 마운트 경로의 파일 소유권을 appuser에 맞게 조정
- **담당 세션**: DevOps
- **상태**: 🔴 미수정

### [Medium] F-007: 비밀번호 최소 길이 과소 (4자)

- **위치**:
  - `app/admin/actions.ts:19` — `if (password.length < 4)`
  - `app/admin/users/actions.ts:25` — `if (password.length < 4)`
- **설명**: 비밀번호 최소 길이가 4자로 매우 짧다. bcrypt cost factor 10은 적절하나, 4자 비밀번호는 사전 공격에 취약. ISMS-P 비밀번호 정책 기준(최소 8자, 영문+숫자 또는 특수문자 조합)에 미달.
- **영향**: 사용자가 단순 비밀번호(예: "1234", "pass") 설정 시 공격자가 짧은 시간 내 크래킹 가능.
- **근거 표준**: ISMS-P 2.5.3 (비밀번호 관리), ISO 27001 A.9.3.1 (비밀 인증 정보 사용)
- **수정 방안**:
  - 최소 8자 이상으로 변경: `if (password.length < 8)`
  - 권장: 영문+숫자 조합 필수 또는 최소 12자 이상 정책 적용
  - 변경 시 기존 사용자에게 다음 로그인 시 비밀번호 변경 강제 기능 검토
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [Low] F-008: 미사용 AWS SDK 의존성 (폐쇄망 정책 위반 가능성)

- **위치**: `package.json` (dependencies 섹션)
  ```json
  "@aws-sdk/client-s3": "^3.1001.0",
  "@aws-sdk/s3-request-presigner": "^3.1001.0"
  ```
- **설명**: AWS SDK 패키지가 추가되어 있으나 `app/`, `lib/` 전체에서 사용 코드가 발견되지 않는다. AWS SDK는 초기화 시 IMDSv2(EC2 메타데이터 서비스), 환경변수 자격증명 등을 탐색하는 동작이 있으며, 폐쇄망에서 불필요한 외부 메타데이터 요청을 시도할 수 있다. 또한 번들 크기를 불필요하게 증가시킨다.
- **영향**: 불필요한 공격 표면 확대, supply chain 위험 증가, EC2 메타데이터 서비스 접근 시도.
- **근거 표준**: ISO 27001 A.12.5.1 (소프트웨어 설치 통제), ISMS-P 2.8.4 (소프트웨어 보안)
- **수정 방안**:
  - 향후 S3 사용 계획이 없다면 즉시 제거: `npm uninstall @aws-sdk/client-s3 @aws-sdk/s3-request-presigner`
  - 향후 사용 계획이 있다면 실제 사용 시점에 추가 (YAGNI 원칙)
- **담당 세션**: DevOps / 백엔드
- **상태**: 🔴 미수정

### [Low] F-009: 라이선스 키(key) 값이 Audit Log details에 평문 기록

- **위치**: `app/licenses/[id]/edit/actions.ts:200`
  ```typescript
  if ((existing.key ?? null) !== (newData.key ?? null))
    changes.key = { from: existing.key, to: newData.key };
  ```
- **설명**: VOLUME 타입 라이선스의 공유 키(key) 값이 변경될 때 변경 전·후 키가 AuditLog.details JSON에 평문으로 저장된다. 라이선스 키가 기밀 자산인 경우 감사 로그 조회 권한이 있는 사용자 모두에게 노출된다.
- **영향**: AuditLog 조회 시 라이선스 키 값 노출. DB 파일 직접 접근 시 키 탈취.
- **근거 표준**: ISO 27701 8.2.4 (민감 데이터 보호), ISMS-P 2.7.1 (암호화 적용)
- **수정 방안**:
  - 키 변경 감사 시 실제 값 대신 "(변경됨)" 또는 키 해시(SHA-256 앞 8자) 기록
  - 예시: `changes.key = { from: existing.key ? "****" : null, to: newData.key ? "****" : null }`
- **담당 세션**: 백엔드
- **상태**: 🔴 미수정

### [Info] F-010: 세션 7일 고정 만료 — 슬라이딩 갱신 없음

- **위치**: `lib/auth.ts:8` — `SESSION_DURATION_MS = 7 * 24 * 60 * 60 * 1000`
- **설명**: 세션이 생성 시 7일 고정 만료로 설정되며 활성 사용 중에도 갱신되지 않는다. 7일 후 활성 사용자가 갑자기 로그아웃됨. 동시에 세션 토큰 탈취 시 7일간 지속 유효.
- **영향**: UX 영향 및 세션 탈취 시 장기간 유효성 유지.
- **근거 표준**: ISO 27001 A.9.4.2, ISMS-P 2.5.5 (세션 관리)
- **수정 방안**:
  - `validateSession` 내에서 만료 시간이 일정 임계치(예: 1일) 이하로 남으면 자동 연장
  - 또는 비활성 시간 기준 만료 검토
- **담당 세션**: 백엔드
- **상태**: 🟡 검토 필요

## 긍정적 발견 사항 (잘 구현된 부분)

- **bcrypt cost 10** — 비밀번호 해싱 적절 (`lib/auth.ts:11`)
- **HttpOnly + SameSite=Lax 쿠키** — 세션 쿠키 보안 설정 적절 (`app/api/auth/login/route.ts:44-50`)
- **비활성 사용자 세션 즉시 파기** — `validateSession`에서 `isActive=false` 즉시 세션 삭제 (`lib/auth.ts:46-49`)
- **Prisma ORM 사용** — SQL Injection 방지 (파라미터 바인딩)
- **트랜잭션 처리** — 복잡한 데이터 변경을 `$transaction`으로 원자적 처리
- **AuditLog 스키마 존재** — 감사 로그 인프라 구축됨 (actor 필드 활성화 필요)
- **.dockerignore 양호** — .env, .git, *.db 등 민감 파일 제외 처리

### 추가 규칙 (백엔드 섹션)

1. **API 라우트 접근 제어**: 데이터 변경(POST/PUT/PATCH/DELETE) API는 반드시 역할 검증을 포함한다 — `requireAdmin()` 또는 `if (user.role !== "ADMIN") return 403`
2. **Server Action 인증**: `"use server"` 함수는 반드시 최상단에서 인증을 확인한다 (`getCurrentUser()` 또는 `requireAdmin()`)
3. **Audit Log actor 필수**: `writeAuditLog()` 호출 시 `actor` 파라미터에 현재 사용자명을 반드시 전달한다
4. **인증 이벤트 로깅 필수**: 로그인 성공/실패, 로그아웃은 AuditLog에 기록한다
5. **비밀번호 최소 8자**: 신규 비밀번호 정책은 최소 8자 이상

### 추가 규칙 (DevOps 섹션)

6. **컨테이너 비root**: dockerfile에 USER 지시문을 추가해 비root 사용자로 실행한다
7. **미사용 의존성 제거**: 실제 사용하지 않는 패키지는 package.json에서 즉시 제거한다
