# 자산 관리 정책 (Asset Management Policy)

> 🔒 보안 세션이 관리
> 준거: ISMS-P 1.1~1.4, ISO 27001 A.8
> 유효기간: 2026-03-05 ~ 2027-03-04

---

## 1. 정책 목표

License Manager의 **정보자산, 시스템자산, 개인정보**를 체계적으로 관리하여:
1. 자산 소재 파악 및 추적
2. 민감도 분류 및 보호 수준 결정
3. 접근 통제 및 권한 관리
4. 생명주기 관리 및 안전한 폐기
5. ISMS-P, ISO 27001, ISO 27701 준수

---

## 2. 적용 범위

### 2.1 적용 대상
- License Manager 애플리케이션의 **모든 자산**
- 사용자: ADMIN, USER, DevOps, Security 역할

### 2.2 제외 사항
- 공개 소스코드 (GitHub Public)
- 공개 라이브러리 (npm, Docker Hub)

---

## 3. 책임과 역할

### 3.1 자산 책임자 (Asset Owner)
| 자산 범주 | 책임자 | 책임사항 |
|---------|--------|---------|
| 라이선스 데이터 | Backend | 데이터 무결성, API 보안 |
| 직원 정보 (개인정보) | HR/Admin | PII 보호, 접근 제한 |
| 라이선스 키 | Backend | 암호화, 마스킹, 보관 |
| 감사 로그 | Security | 로그 수집, 분석, 보존 |
| 시스템/인프라 | DevOps | 서버 보안, 백업, 복구 |
| 접근 권한 | Security | RBAC 관리, 검증 |

### 3.2 역할별 책임

#### Backend
- [ ] 라이선스/직원 데이터 입력 검증
- [ ] PII 필드 접근 로그 기록
- [ ] 라이선스 키 암호화 저장
- [ ] API 역할 검증 (RBAC)

#### Frontend
- [ ] 개인정보 화면 표시 최소화
- [ ] 민감 정보 localStorage/sessionStorage 미저장
- [ ] XSS 방지

#### DevOps
- [ ] DB 백업 암호화
- [ ] 접근 권한 최소화 (chmod 600)
- [ ] Docker 보안 설정 (비root 실행)
- [ ] 로그 수집 및 보존

#### Security (보안 담당)
- [ ] 자산 인벤토리 유지
- [ ] 민감도 재평가 (분기별)
- [ ] 감사 로그 분석
- [ ] 리포팅 (월간/분기별/연간)

## 4. 자산 분류 기준

### 4.1 민감도 레벨

| 레벨 | 설명 | 예시 | 보호조치 |
|------|------|------|---------|
| **Critical** | 공개 시 심각한 피해 | 라이선스 키, 비밀번호 | 암호화, 최소 권한 접근 |
| **High** | 개인정보, 비즈니스 위험 | 직원명, 이메일 | 접근 제한, 감사로그 |
| **Medium** | 비즈니스 데이터 | 라이선스, 할당 정보 | 접근 제한, 감시 |
| **Low** | 조직 정보 | 조직도, 그룹 정보 | 기본 접근 제어 |

### 4.2 분류 프로세스

```
[새로운 자산 발견]
    ↓
[민감도 평가]
  - 공개 시 피해도
  - 개인정보 포함 여부
  - 비즈니스 가치
    ↓
[레벨 결정] (Critical/High/Medium/Low)
    ↓
[보호조치 결정]
  - 접근 통제
  - 암호화 필요성
  - 감사 로그
  - 보유 기간
    ↓
[인벤토리 등록]
```

## 5. 자산 생명주기 관리

### 5.1 등록 (Registration)

**시점**: 새로운 자산 개발/도입 시
**담당**: 해당 백엔드/DevOps
**절차**:
1. 자산명, 설명, 위치 기록
2. 민감도 평가 (Security와 협의)
3. 책임자 지정
4. 접근 통제 정의
5. `tasks/security/asset-inventory.md` 업데이트

**체크리스트**:
- [ ] 자산명 정의됨
- [ ] 민감도 레벨 결정됨
- [ ] 책임자 지정됨
- [ ] 보유 기간 정의됨

### 5.2 분류 (Classification)

**시점**: 등록 후 또는 분기 검토 시
**담당**: Security
**절차**:
1. 민감도 재평가
2. 비즈니스 가치 평가
3. 위험도 평가 (위협 모델 기반)
4. 보호 수준 결정

**분류 결과**:
- `tasks/security/asset-inventory.md` 의 "민감도" 컬럼

### 5.3 운영 (Operation)

**주기**: 지속적 (월간 검토, 분기별 검증)
**담당**: 자산 책임자 + Security
**활동**:

#### 월간 (Monthly)
- [ ] 자산 상태 확인 (정상 운영 여부)
- [ ] 접근 이상 징후 모니터링 (AuditLog 기반)
- [ ] 백업 실행 (DB 및 설정)

#### 분기별 (Quarterly)
- [ ] 자산 인벤토리 갱신
- [ ] 접근권한 검증
  - ADMIN 역할이 필요한가?
  - 미사용 계정 비활성화
  - 권한 증가 사항 검토
- [ ] 민감도 재평가
- [ ] 정책 준수 확인

#### 연간 (Annually)
- [ ] 전체 자산 감시
- [ ] 폐기 대상 식별
- [ ] 정책 효과성 평가
- [ ] 컴플라이언스 보고서 작성

### 5.4 검토 (Review)

**시점**: 분기별, 연간
**담당**: Security + 자산 책임자
**내용**:
- 자산이 여전히 필요한가?
- 민감도 변경 필요한가?
- 보호 조치가 효과적인가?
- 정책 위반 사항이 있는가?

**결과**:
- 계속 운영 → 5.3 으로 복귀
- 폐기 대상 → 5.5 로 진행

### 5.5 폐기/보관 (Disposal)

**시점**: 자산 수명 종료 또는 비용 절감 필요 시
**담당**: DevOps (기술), HR/Admin (정책)
**절차**:

#### 데이터 폐기
1. 안전한 삭제 실행 (overwrite, encryption key removal)
2. 폐기 증명서 발급
3. 감사 로그 기록

#### 로그 보존
- 일부 감사 로그는 법적 요구 기간(2년) 보존
- 보존 로그는 접근 제한, 암호화 저장

#### 문서화
- 폐기 날짜, 방법, 증명서
- `tasks/security/asset-inventory.md` 기록

**체크리스트**:
- [ ] 데이터 백업 완료
- [ ] 안전한 삭제 실행
- [ ] 감사 로그 기록
- [ ] 폐기 증명서 발급
- [ ] 인벤토리 업데이트

## 6. 자산별 보호 조치

### 6.1 정보자산 (Information Assets)

#### 라이선스 데이터 (License, LicenseSeat)
**민감도**: Critical / High
**보호 조치**:
- [x] API RBAC 검증 (Admin만)
- [x] 라이선스 키 마스킹 (감사로그)
- [ ] 라이선스 키 암호화 저장 (★ 미구현)
- [x] 접근 로그 기록 (AuditLog)

**필요 조치**:
- 데이터베이스 암호화 필드 추가
- 실시간 감시 (접근 이상 탐지)

#### 개인정보 (Employee)
**민감도**: High
**보호 조치**:
- [x] API RBAC 검증
- [x] 접근 로그 기록 (AuditLog + actor)
- [ ] PII 필드 암호화 저장 (★ 미구현)
- [ ] 보유 기간 자동 파기 스케줄러 (★ 미구현)

**필요 조치** (우선순위):
1. PII 암호화 저장 (2026-04)
2. 자동 파기 배치 (2026-05)
3. 접근 제한 강화 (READ 권한만)

#### 감사 로그 (AuditLog)
**민감도**: Critical
**보호 조치**:
- [x] Admin 전용 접근
- [x] 모든 변경사항 기록 (actor, timestamp, details)
- [x] 민감 정보 마스킹 (라이선스 키)
- [ ] 로그 무결성 검증 (★ 미구현)
- [ ] 로그 암호화 저장 (★ 미구현)

**필요 조치**:
- 로그 서명 (HMAC) 추가
- 로그 백업 암호화

### 6.2 시스템 자산 (System Assets)

#### 애플리케이션 (Next.js App)
**보호 조치**:
- [x] 입력 검증 (타입, 길이)
- [x] XSS 방지 (React 기본)
- [x] SQL Injection 방지 (Prisma ORM)
- [x] Rate Limiting (로그인)
- [ ] HTTPS (★ 폐쇄망 환경)

#### 데이터베이스 (SQLite)
**보호 조치**:
- [x] 파일 접근 권한 제한 (chmod 600)
- [x] 트랜잭션 기반 일관성 보장
- [ ] 데이터베이스 암호화 (★ 미구현)
- [ ] 자동 백업 (★ 미구현)
- [ ] 백업 암호화 (★ 미구현)

**필요 조치** (우선순위):
1. 자동 백업 시스템 구축 (2026-03)
2. 백업 암호화 (2026-03)
3. 복구 테스트 (월간)

#### 인프라 (EC2, Docker)
**보호 조치**:
- [x] 폐쇄망 (VPN 기반)
- [x] .dockerignore (민감 파일 제외)
- [ ] 컨테이너 비root 실행 (★ 미구현)
- [ ] 이미지 스캔 (취약점 검사) (★ 미구현)

**필요 조치**:
1. Dockerfile USER 지시문 추가 (2026-03)
2. 정기 이미지 스캔 (월간)

## 7. 접근 통제 (Access Control)

### 7.1 역할 기반 접근 통제 (RBAC)

| 자산 | ADMIN | USER | DevOps | Security |
|------|-------|------|--------|----------|
| License | R/W | R | R | R |
| Employee | R/W | R (own) | R | R |
| LicenseSeat | R/W | - | R | R |
| AuditLog | R/W | - | R | **R/W** |
| User 계정 | R/W | - | - | R/W |
| System Config | R/W | - | **R/W** | R |

### 7.2 접근 권한 검증 (분기별)

**절차**:
1. 현재 사용자 목록 확인
2. 각 사용자의 역할 검증
3. 권한 확대 사항 검토
4. 미사용 계정 비활성화
5. 변경사항 기록

**체크리스트**:
- [ ] ADMIN 역할 필요성 검증
- [ ] USER 역할 권한 제한 확인
- [ ] 미사용 계정 비활성화
- [ ] 신규 계정 승인

## 8. 지속적 리포팅 (Continuous Reporting)

### 8.1 월간 리포트 (Monthly Report)

**일정**: 매월 5일
**담당**: Security
**내용**:
- 자산 현황 요약 (개수, 상태)
- 접근 이상 현황 (비정상 로그인, 권한 변경)
- 백업 상태
- 규정 위반 사항
- 개선 필요사항

### 8.2 분기별 리포트 (Quarterly Report)

**일정**: 3월, 6월, 9월, 12월 15일
**담당**: Security + 자산 책임자
**내용**:
- 자산 인벤토리 갱신 결과
- 민감도 재평가 결과
- 접근권한 검증 결과
- 정책 준수 현황
- Risk Assessment 결과
- 권장사항

### 8.3 연간 감사 (Annual Audit)

**일정**: 1월 31일
**담당**: Security + 외부 감사인 (선택사항)
**내용**:
- 전체 자산 감시 보고서
- ISMS-P 준수 현황 (101개 항목)
- ISO 27001 A.8 준수 현황
- ISO 27701 PII 관리 현황
- 개선 로드맵
