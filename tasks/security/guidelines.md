# 보안 개발 가이드라인

> 이 파일은 🔒 보안 세션이 관리한다.
> 프론트엔드, 백엔드, DevOps 세션은 코드 작성 전 이 파일을 확인하고 규칙을 준수해야 한다.
> 최종 업데이트: (보안 리뷰 후 날짜 기입)

## 준거 표준
- ISMS-P (KISA 인증 기준)
- ISO/IEC 27001:2022 (정보보안경영시스템)
- ISO/IEC 27701:2019 (개인정보보호경영시스템)
- Zero Trust Architecture (NIST SP 800-207)

---

## 공통

### 입력/출력 (ISO 27001 A.14.2 / ISMS-P 2.8)
- 외부 입력은 항상 검증한다 (타입, 길이, 형식, 허용 범위)
- 에러 응답에 스택 트레이스, 내부 경로, DB 쿼리를 포함하지 않는다
- 로그에 비밀번호, 토큰, 개인정보를 출력하지 않는다

### Zero Trust 원칙 (전 세션 공통)
- 모든 요청은 인증·인가를 거친다 (Verify Explicitly)
- 최소 권한만 부여한다 — 기본값은 "거부" (Least Privilege)
- 이미 침해당한 상태를 가정하고 설계한다 (Assume Breach)

## 프론트엔드

### XSS 방지 (ISO 27001 A.14.2 / ISMS-P 2.8.1)
- 사용자 입력을 innerHTML 또는 dangerouslySetInnerHTML로 렌더링하지 않는다
- API 응답의 에러 메시지를 그대로 화면에 표시하지 않는다
- URL 파라미터를 DOM에 직접 삽입하지 않는다

### 데이터 보호 (ISO 27701 7.4 / ISMS-P 3.3)
- 민감 정보를 localStorage/sessionStorage에 저장하지 않는다
- 개인정보를 콘솔에 출력하지 않는다 (console.log 주의)
- 폼 제출 시 클라이언트 검증과 별개로 서버 검증을 신뢰한다

### 인증 (Zero Trust: Verify Explicitly)
- 인증 상태를 클라이언트에서만 판단하지 않는다 (서버 확인 필수)
- 세션 만료 시 적절한 리다이렉션 처리

## 백엔드

### 인증/인가 (ISO 27001 A.9.4 / ISMS-P 2.5 / Zero Trust)
- 모든 API 라우트에 인증 확인을 적용한다 (getCurrentUser 또는 requireAdmin)
- 요청마다 세션을 재검증한다 (캐시된 인증 상태에 의존하지 않음)
- 비밀번호는 bcryptjs로 해싱한다 (cost factor 10 이상)
- 세션 쿠키: HttpOnly=true, SameSite=Lax 필수
- 로그인 실패 시 실패 횟수 제한 및 로깅

### 접근 통제 (ISO 27001 A.9.1 / ISMS-P 2.6 / Zero Trust: Least Privilege)
- 권한이 필요한 작업은 소유권/역할을 반드시 확인한다 (IDOR 방지)
- API별 역할 기반 접근 통제 (RBAC) 적용
- 관리자 기능과 일반 사용자 기능의 접근 경로를 분리한다

### 입력 검증 (ISO 27001 A.14.2 / ISMS-P 2.8)
- SQL/ORM 쿼리에 사용자 입력을 직접 삽입하지 않는다 (Prisma 파라미터 바인딩 사용)
- 파일 경로를 받는 API에서 Path Traversal을 방지한다 (../ 필터링)
- 요청 body, query, params 모두 검증한다

### 개인정보보호 (ISO 27701 7.4, 8.2 / ISMS-P 3.1~3.5)
- Employee 테이블의 개인정보 항목을 식별하고 분류한다
- 개인정보는 필요 최소한만 수집한다 (데이터 최소화)
- 개인정보 조회/수정/삭제 시 감사 로그를 기록한다
- 보유 기간이 만료된 개인정보는 안전하게 파기한다
- 개인정보를 포함하는 API 응답에서 불필요한 필드를 제외한다

### 로깅/감사 (ISO 27001 A.12.4 / ISO 27701 8.2.5 / ISMS-P 2.11)
- 인증 이벤트를 로깅한다 (로그인 성공/실패, 로그아웃)
- 관리자의 데이터 수정/삭제 행위를 로깅한다
- 로그에 민감 정보(비밀번호, 개인정보)를 포함하지 않는다
- 로그 보존 기간을 정의하고 준수한다

## DevOps

### 컨테이너 보안 (ISO 27001 A.12.6 / ISMS-P 2.10)
- Docker 이미지에 .env, .git, node_modules 포함하지 않는다 (.dockerignore 확인)
- 컨테이너를 root 사용자로 실행하지 않는다 (가능한 경우)
- 이미지에 불필요한 도구(curl, wget 등)를 포함하지 않는다
- 베이스 이미지를 정기적으로 업데이트한다

### 시크릿 관리 (ISO 27001 A.10.1 / ISMS-P 2.7)
- 환경변수에 시크릿을 하드코딩하지 않는다 (docker-compose.yml에 평문 금지)
- 시크릿은 환경변수 또는 시크릿 매니저로 주입한다
- Git에 시크릿을 커밋하지 않는다

### 네트워크 (ISO 27001 A.13.1 / Zero Trust: 마이크로 세그멘테이션)
- 사용하지 않는 포트를 외부에 노출하지 않는다
- 폐쇄망의 이점을 활용하되, 내부 네트워크도 신뢰하지 않는다
- DB 파일 접근 권한은 최소한으로 설정한다 (chmod 600)

### DB 보안 (ISO 27001 A.12.3 / ISO 27701 8.2.4)
- DB 백업 시 암호화 적용
- 백업 파일 접근 권한 제한
- DB 파일 무결성 검증 주기적 실행
